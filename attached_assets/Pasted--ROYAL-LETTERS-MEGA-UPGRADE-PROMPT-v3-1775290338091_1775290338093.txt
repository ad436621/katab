── 1A. SERVICE WORKER REWRITE (public/sw.js) ─────────────

Replace the existing service worker with a production-grade
implementation:

INSTALL event:
  Pre-cache the app shell:
    '/', '/index.html', '/manifest.json',
    all hashed JS/CSS bundles (use self.__WB_MANIFEST if using
    workbox, otherwise hardcode the shell files),
    notification sound file '/sounds/notify.mp3',
    offline fallback page '/offline.html'
  self.skipWaiting() to activate immediately.

ACTIVATE event:
  clients.claim() to take control of all open tabs.
  Delete old caches from previous versions.

FETCH event:
  - API calls (/api/*): Network-first, fall back to a cached
    JSON error response if offline.
  - Static assets (JS/CSS/images): Cache-first.
  - HTML navigation requests: Network-first, fall back to
    cached '/index.html', then to '/offline.html'.

PUSH event (WhatsApp-style):
  Parse the push payload JSON:
    { type, title, body, icon, badge, tag, url,
      recipientName, letterId, sound, vibrate }

  Show notification with:
    title, body, icon: '/icons/icon-192.png',
    badge: '/icons/badge-72.png' (monochrome icon for status bar),
    tag: use letterId or type as tag (collapses duplicates),
    renotify: true (always show even if same tag),
    requireInteraction: true for type 'new_reply'
      (stays until dismissed, like WhatsApp messages),
    silent: false,
    vibrate: [200, 100, 200] (double-pulse like WhatsApp),
    data: { url, letterId, type },
    actions:
      For type 'new_reply':
        [ { action: 'open', title: '📖 Open Letter' },
          { action: 'dismiss', title: '✕ Dismiss' } ]
      For type 'letter_read':
        [ { action: 'open', title: '👁 View Letter' } ]
      For type 'message_unlocked':
        [ { action: 'open', title: '🔓 Open Now' },
          { action: 'dismiss', title: 'Later' } ]

NOTIFICATIONCLICK event:
  event.notification.close()
  If action === 'dismiss': return.
  If action === 'open' or default click:
    const url = event.notification.data.url
    Focus existing open window at that URL if one exists,
    otherwise open a new window: clients.openWindow(url)

NOTIFICATIONCLOSE event:
  Track dismissals — POST /api/notifications/dismissed with
  the notification tag so the server knows it was seen.

MESSAGE event (from main thread):
  Handle { type: 'SKIP_WAITING' } → self.skipWaiting()
  Handle { type: 'BADGE_COUNT', count } →
    if (navigator.setAppBadge) navigator.setAppBadge(count)
  Handle { type: 'CLEAR_BADGE' } →
    if (navigator.clearAppBadge) navigator.clearAppBadge()

── 2B. FRONTEND WEBSOCKET HOOK ───────────────────────────

Create src/hooks/useAdminSocket.ts:

  export function useAdminSocket(onEvent: (e: WsEvent) => void) {
    useEffect(() => {
      // Connect to ws:// or wss:// same origin
      const ws = new WebSocket(
        `${location.protocol === 'https:' ? 'wss' : 'ws'}://${location.host}/ws`
      )
      ws.onmessage = (msg) => {
        const event = JSON.parse(msg.data)
        onEvent(event)
      }
      // Exponential backoff reconnect on close/error
      // Start at 1s, double each attempt, cap at 30s
      // Reset backoff on successful open
      return () => ws.close()
    }, [])
  }

Encryption: Every new TEXT field in the DB that is user
  content MUST be encrypted with AES-256-GCM using the
  existing encrypt/decrypt helpers. No exceptions.

Error handling: Every new Express route is wrapped in
  try/catch and calls next(err) on failure.

Mobile-first: Every new UI component has base styles for
  320px and scales up. No new horizontal overflow introduced.
  All new interactive elements min 44px touch target.

Migrations: After every DB schema change, run the Drizzle
  migration. Never manually alter the DB. Always update the
  Drizzle schema file in lib/db/ first.

Sonner toasts: ALL user-facing feedback uses sonner toast.
  No window.alert(). No console.error() shown to users.

TypeScript: No `any` types. Define interfaces for all
  new API request/response shapes.

No regressions: After implementing each phase, verify that
  all previously working features still work:
  login, compose, verify flow, replies, scheduled messages,
  push subscriptions, image upload, settings page.
